01总框架：一切归结为「怎么看」和「怎么动」

无论工具名字多花哨，差异本质只有两个维度：感知从哪来、动作打到哪一层。

范式怎么看（感知）怎么动（动作）作用层
① 像素视觉屏幕截图（像素）模拟鼠标/键盘坐标点击整个屏幕 / 任意 GUI
② DOM/CDPHTML DOM 结构树选择器 / CDP 协议命令浏览器内部
③ 无障碍树OS 语义控件树 (a11y)控件级 invoke桌面应用 + 网页
④ API/MCP结构化数据 / 工具 schema直接调函数绕过 UI
⑤ 混合(视觉+DOM)截图 + DOM 标注框定位后执行主要在浏览器
✓核心判断（已验证 · 跨工具综合）
通用性与可靠性/成本成反比。像素视觉最通用（任何界面都能开），但最慢、最难定位、可靠性最低；越往 DOM/API 下沉，越快、越准、越便宜，但适用面越窄。这条规律贯穿全文，也是第 07 节决策树的地基。

③ 无障碍树 & ④ API/MCP 的定位

无障碍树介于像素与 DOM 之间——读 OS 为读屏软件暴露的语义控件树（Windows UIA / macOS AX / Web ARIA），拿到「这是个名为『提交』的按钮」。比截图省 token、比 DOM 跨得更广（覆盖原生桌面），但很多自绘 UI / Canvas / 游戏 a11y 信息缺失。

a11y 不止浏览器：桌面原生应用同样能读

Demo 里的 a11y 走的是浏览器 CDP getFullAXTree。同一范式在桌面 OS 有独立底层 API：Windows UIA / macOS AXUIElement / Linux AT-SPI——都是给读屏软件（NVDA/VoiceOver）用的语义树，App 只要支持无障碍就免费暴露给 Agent。

本机用 Python uiautomation 在真实系统计算器上实测：读控件树 → 按控件名点按钮算 7×8 → 读回结果，全程零屏幕坐标。真实输出：

  ButtonControl  Multiply by
TextControl    Display is 0        ← 结果控件，语义属性直接可读

API/MCP 不是「操作电脑」的竞品，而是天花板对照组：能用接口就别去点 UI。MCP 让「为每个集成写死胶水」变成「实现一次、接入生态」。它的存在也提醒我们——UI 操作范式的全部意义，是为「没有 API 的那部分世界」兜底。

三条压舱石判断 ✓ 已验证

1. 像素视觉 = 最通用但最弱。任何界面都能开，但慢、贵、grounding 难、可靠性最低（OSWorld 发布期 <25% vs 人类 ~72%）。

2. DOM/CDP = 最快最便宜最准，但只在浏览器且怕改版——除非用 AI 中介（如 Skyvern）化解选择器脆性。

3. 混合 = 截图做 grounding + DOM 抽可点元素（Set-of-Marks / Vision-LLM 映射），用算力换鲁棒性。

03a11y 范式：浏览器 vs 桌面 OS

同一棵语义树，两个入口。Demo 里读的是浏览器内的 AX 树；换到桌面原生应用，走的是 OS 无障碍 API——role/name 结构完全同构。

无障碍树不是浏览器发明的，它源自操作系统给读屏软件（Windows 的 NVDA/JAWS、macOS 的 VoiceOver、Linux 的 Orca）设计的标准接口。应用只要支持读屏（大厂 App 基本都合规实现），就等于免费给了 Agent 一棵语义树——拿到的是 {role:"button", name:"发送", enabled:true} 这样的结构化控件，而非像素。

┌─────────────────────────────────────┐ │ AI Agent │ └──────────┬──────────────┬────────────┘ │ │ 浏览器内的网页 桌面原生应用 │ │ CDP Accessibility 域 OS 无障碍 API (Chromium 自己的 AX 树) ├─ UIA (Windows) │ ├─ AX (macOS) │ └─ AT-SPI (Linux) │ │ getFullAXTree ← 拉整棵树 遍历 ControlType / AXRole 树 └──────┬───────┘ role + name，两边同构 （浏览器也把网页 AX 树桥接给 OS UIA）

三大桌面平台的无障碍 API

平台API说明
WindowsUI Automation (UIA)微软官方，取代旧 MSAA。Win32/WPF/UWP/Electron 都支持，生态最完善。工具：pywinauto、FlaUI、Power Automate Desktop
macOSAccessibility (AX)AXUIElement 体系，VoiceOver 就靠它。权限门槛高——须显式授权「辅助功能」+「屏幕录制」
LinuxAT-SPI2通过 D-Bus 暴露，GNOME/GTK/Qt 应用支持。库：pyatspi

有意思的是：浏览器本身也是个 OS 应用。Chrome 会把网页的 AX 树桥接到 Windows UIA——所以理论上用 UIA 也能读到网页内容，只是没有 CDP 直接读浏览器内部树那么快、那么全。

桌面 a11y 的优劣（对照报告 §02）

✓ 优势：语义化（按名字/角色操作，不点像素）；稳（窗口移动/换主题/换分辨率都不影响）；省 token；能拿到隐藏属性（enabled/checked/value）。

✗ 劣势：强依赖应用是否好好实现无障碍——自绘 UI（游戏、部分国产软件、Electron 里的 canvas、Flutter 早期）几乎是空树；完整树很嘈杂（和 §05 实测「a11y 全树最费 token 13812」同一个坑）；macOS 权限门槛高；跨平台要写三套。

适用：标准控件为主的传统桌面软件（Office、系统设置、Win32 表单、浏览器）。
不适用：自绘 / 游戏 / 远程桌面画面 / 图片内文字 → 只能退回像素视觉范式（这正是 Claude CU、UI-TARS 的立身之本：它不挑应用是否实现无障碍）。

一句话选型：应用规规矩矩用系统控件 → 桌面 a11y（UIA 等）首选，又快又稳又省 token；应用自绘 / 没实现无障碍 / 是画面 → 只能上视觉截图。实战里成熟 GUI Agent（UI-TARS、Operator）常混用：先读 a11y 树拿结构，读不到的区域再用视觉兜底。上一节那张桌面计算器实测卡片（👁 看代码）就是 Windows UIA 的完整跑通证据。

送入 LLM 的 token 成本（越低越好）

这是最能拉开差距的指标——直接决定每次调用的钱和延迟。

★ CDP 在浏览器内跑 DOM API，只回传精炼结果，token 最低。DOM 正则若把整页 HTML 喂给 LLM 达 8689 tokens，但本地裁剪后可降到 291（图中虚线标注）——「喂原始 HTML」是最常见的浪费。

端到端延迟（越低越好）

网络两路与 CDP 都在 ~1s 级；视觉因需启动渲染+截图显著更慢。

注：DOM 正则最快（815ms）因为它只是 fetch 纯文本、不启动浏览器；但它把脆性和 token 成本转嫁到了下游。

网络 payload（传输字节，越低越好）

像素视觉的 PNG 截图是最重的载荷；CDP 只回传抽取结果因而最轻。

对数视角更公平：视觉 124KB 是 CDP 1.2KB 的 约 107×。

⚠三个反直觉的真实发现
1. CDP 页内提取是隐藏赢家——token 最低、payload 最小，因为它在浏览器里跑 DOM API 只回传结果。
2. a11y 全树竟然最费 token（13812）——完整 AX 树很嘈杂，不裁剪比整页 HTML 还贵，修正了「a11y 一定省 token」的直觉。
3. 视觉单张截图 token（1536）反而低于未裁剪文本树——这解释了为何当目标结构混乱时，视觉范式反而有吸引力。

时效性

所有基准数字为 2024–2025 初快照。Claude 14.9%/22%、UI-TARS-72B 24.6%、最佳模型 12.24% 均为发布期。站在 2026-07，OSWorld/WebVoyager/WebArena 的 SOTA 已明显更高、人机差距大幅收窄。范式间相对优劣结论仍成立，具体百分比会过时。

实测的边界

Demo 数据是单次本机运行，延迟受网络波动影响；token 为字符数/4 的估算（视觉按 (w×h)/750）。它证明的是范式间的数量级差异，非精确基准。脚本 bench-all.mjs 已随报告落盘，可复跑。
